Design System

A walkthrough of how a rough Figma style guide became a true component library — and how AI agents now keep it growing.

When I joined, the team had a rough style guide in Figma — a reference file of shared elements, with no real components or tokens. I co-led a full revamp with our other designer: turning the guide into a proper library with proper component properties, expanding from mobile to web, and most recently, building AI agents that automate component creation against our tokens and HIG/Material guidelines.

Mobile

Mobile foundations
& 13 components

Implemented the iOS & Android library from the ground up — type ramp, 4‑pt spacing scale, and 13 componentized primitives totaling 113 variants. Every fill, stroke, and dimension binds to a semantic token, so light/dark themes and platform variants flow through the whole library from a single source.

Two-Agent Flow

Automate DS creation
& alteration

A 2-designer team can’t scale linearly with the product. So I built a pair of agents — builder.md and evaluator.md — that automate the repetitive half of design system work: creating new components against our tokens, and altering existing ones without breaking conventions.

builder.md
Agent · Builder
Creates & updates components
Drafts new components or extends existing ones using our token set and the property conventions of the library.
Bind variables to fills, type, spacing
Set up variants, boolean, instance swap, text properties
Follow the library’s naming & structure
evaluator.md
Agent · Evaluator
Reviews against the rules
Audits a component or screen against our tokens and the platform guidelines — HIG for iOS, Material for Android — and reports what’s off.
Flag raw hex / off-token values
Check spacing, hit areas, type ramp
Surface HIG / Material conflicts
design brief → builder.md → component draft → evaluator.md → designer review & ship
